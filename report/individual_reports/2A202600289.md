## I. Technical Contribution (15 Points)

**Modules Implemented**: `src/tools/u1.py`

**Mô tả đóng góp:**  
Tôi implement **UC1 — Crypto Price Tool**, là tool đầu tiên và cơ bản nhất trong pipeline ReAct Agent. Tool này có nhiệm vụ lấy giá real-time của một đồng crypto bất kỳ thông qua DIA API (https://api.diadata.org).

**Code Highlights** — `src/tools/u1.py`:

```python
def get_crypto_price(symbol="BTC"):
    """
    Tool cho Agent: Lấy giá của một đồng coin bất kỳ qua DIA API.
    """
    url = f"https://api.diadata.org/v1/quotation/{symbol}"

    try:
        response = requests.get(url)
        response.raise_for_status()
        data = response.json()

        normalized_result = {
            "symbol": data.get("Symbol"),
            "name": data.get("Name"),
            "price_usd": round(data.get("Price", 0), 2),
            "price_yesterday": round(data.get("PriceYesterday", 0), 2),
            "last_updated": data.get("Time")
        }
        return normalized_result

    except Exception as e:
        return f"Lỗi: Không lấy được dữ liệu cho {symbol}. Chi tiết: {e}"
```

**Điểm đáng chú ý trong thiết kế:**
- API của DIA đã normalize dữ liệu sẵn (field `Price`, `PriceYesterday`, `Symbol`, `Name`, `Time`), nên không cần xử lý phức tạp.
- Output được normalize thành dict với các key nhất quán (`price_usd`, `price_yesterday`, `last_updated`), giúp các tool downstream (UC4 `aggregate_crypto_summary`, UC5 `evaluate_investment`) dễ consume.
- Graceful degradation: nếu API lỗi (network, HTTP error, JSON parse), tool trả về string lỗi thay vì crash agent.

**Cách tool tích hợp vào ReAct loop:**  
Tool được đăng ký vào `ToolRegistry` trong `registry.py` với tên `get_crypto_price`:
```python
registry.register(
    name="get_crypto_price",
    func=get_crypto_price,
    description="Lấy giá real-time của một đồng crypto (VD: BTC, ETH, SOL).",
    parameters={"symbol": "Ký hiệu coin, mặc định 'BTC'"},
    required_params=[],
)
```
Khi agent sinh ra `Action: get_crypto_price(symbol='BTC')`, `ToolRegistry.execute()` parse argument và gọi hàm này. Kết quả được serialize thành JSON và đưa vào `Observation` của bước tiếp theo trong ReAct loop.

## II. Debugging Case Study (10 Points)

**Problem Description:**  
Khi khởi chạy lần đầu (session `09:22:19`), agent crash ngay ở **step 1** với lỗi `UnicodeEncodeError` — không gửi được request lên OpenAI API:

```
LLM_ERROR at step 1: 'utf-8' codec can't encode characters in position 1796-1797: surrogates not allowed
UnicodeEncodeError: 'utf-8' codec can't encode characters in position 1796-1797: surrogates not allowed
```

**Log Source** — `logs/2026-04-06.log`, dòng 10–28:
```json
{"timestamp": "2026-04-06T09:22:38.933314", "event": "AGENT_STEP", "data": {"step": 1}}
LLM_ERROR at step 1: 'utf-8' codec can't encode characters in position 1796-1797: surrogates not allowed
...
UnicodeEncodeError: 'utf-8' codec can't encode characters in position 1796-1797: surrogates not allowed
```

**Diagnosis:**  
Nguyên nhân nằm ở tầng serialization của OpenAI SDK (`openai/_utils/_json.py`). Input từ user chứa **surrogate characters** (ký tự Unicode không hợp lệ, thường phát sinh do terminal Windows encode sai khi nhập tiếng Việt — ví dụ `\udcc3`). Chuỗi này được đưa vào `messages` của API call, và `json.dumps()` fail khi gặp surrogate.

Lỗi này không phải do prompt hay tool spec, mà do **vấn đề encoding ở tầng transport** — đặc thù của Windows terminal (PowerShell/CMD mặc định dùng codepage 1252 thay vì UTF-8).

**Solution:**  
Thêm bước sanitize input trước khi gửi lên LLM, sử dụng `encode/decode` với `errors='replace'` để loại bỏ surrogate:
```python
# Trong registry.py (đã được áp dụng cho tool output)
output.encode("utf-8", errors="replace").decode("utf-8")
```
Sau khi fix, các session từ `09:26:07` trở đi hoạt động bình thường — agent hoàn thành full ReAct loop (3 bước) với `AGENT_END: {"steps": 3, "has_answer": true}`.

## III. Personal Insights: Chatbot vs ReAct (10 Points)

**1. Reasoning — Thought block giúp ích như thế nào?**

Block `Thought` buộc agent phải *lên kế hoạch trước khi hành động*. Quan sát từ log session `09:26:29`, khi user hỏi "tôi có nên mua bitcoin vào cuối tuần này ko?", agent không trả lời ngay mà suy luận:

> *"Để đưa ra gợi ý mua Bitcoin vào cuối tuần này, tôi cần thu thập dữ liệu tổng hợp về giá hiện tại, xu hướng giá gần đây và tin tức liên quan..."*

Sau đó agent tự động gọi 3 tool liên tiếp (`aggregate_crypto_summary` → `evaluate_investment`) và tổng hợp kết quả. Một chatbot thông thường sẽ trả lời bằng kiến thức cũ hoặc đoán mò — không có dữ liệu real-time.

**2. Reliability — Khi nào Agent tệ hơn Chatbot?**

Agent thực sự xử lý kém hơn trong hai trường hợp quan sát được:

- **Out-of-scope query đơn giản**: Khi user hỏi *"tôi nên mua gà ko?"* (log `09:27:03`), agent detect đúng `AGENT_OUT_OF_SCOPE` và từ chối. Tuy nhiên, một chatbot có thể trả lời vui vẻ và hữu ích hơn trong ngữ cảnh này.

- **Tool name case-sensitive bug**: Ở session `09:27:22`, agent gọi `Fetch_latest_news` (chữ hoa F) thay vì `fetch_latest_news` → `ToolNotFound`. Agent phải mất thêm 1 vòng lặp để sửa lỗi. Chatbot không có vấn đề này vì không gọi tool.

**3. Observation — Feedback từ environment ảnh hưởng thế nào?**

Observation hoạt động như *working memory* của agent. Ví dụ trong session `09:48:12`:

- **Step 1**: Observation từ `get_crypto_price` → BTC = $69,854.53, +4.28% 24h
- **Step 2**: Agent dùng kết quả này để quyết định tiếp tục gọi `get_price_trend(days=30)`
- **Step 3**: Observation trend UP → agent gọi tiếp `fetch_latest_news`
- **Step 4**: Observation đầy đủ → gọi `evaluate_investment` → kết luận **Buy** với confidence 0.63

Nếu bất kỳ tool nào trả về lỗi (ví dụ `NEWS_API_KEY not found` ở session `09:37:22`), agent điều chỉnh ngay bước tiếp theo — đây là điểm mạnh so với chatbot cứng nhắc.

## IV. Future Improvements (5 Points)

- **Scalability**: Chuyển mỗi tool call thành **async task** (dùng `asyncio` hoặc `concurrent.futures`). Tool `aggregate_crypto_summary` hiện gọi tuần tự `get_crypto_price` → `get_price_trend` → `fetch_latest_news`; nếu chạy song song có thể tiết kiệm ~2 giây/request. Với production traffic cao, thêm **task queue** (Celery + Redis) để xử lý nhiều agent session đồng thời.

- **Safety**: Thêm một **Supervisor LLM** (lightweight model) chạy song song để audit action của agent trước khi execute — đặc biệt quan trọng nếu mở rộng sang các tool có side-effect (giao dịch thực, gửi alert). Ngoài ra, cần rate-limit tool calls để tránh abuse API (hiện log cho thấy có session trùng lặp call cùng tool x7 lần do bug concurrency).

- **Performance**: Với hệ thống nhiều tool (>20), việc LLM chọn tool bằng description string sẽ kém hiệu quả. Giải pháp: dùng **Vector DB** (Chroma, Faiss) để embed tool descriptions và retrieve top-k tool liên quan nhất với query của user — giảm context size và cải thiện độ chính xác tool selection.
